# Dashboard Spec — Real-time Skill Dashboard

## Current state

CrewLoop has no runtime observability component. Skills are static Markdown files consumed by the agent. The only web presence is the Docusaurus documentation site (`docs/`), which is static and unrelated to live agent execution.

## Desired state

A new runtime component under `servers/dashboard/` provides a local web dashboard that streams agent activity in real time. It consists of:

1. **Event server** (`servers/dashboard/src/server.ts`) — receives normalized events via HTTP, maintains session state, broadcasts updates via WebSocket.
2. **Shim** (`servers/dashboard/src/shim.ts`) — reads agent hook JSON from stdin, filters sensitive data, normalizes to a common event schema, and POSTs to the event server.
3. **Agent adapters** — configuration and small wrappers for Kimi Code, Codex CLI, and OpenCode.
4. **Skill inference engine** (`servers/dashboard/src/skills/infer.ts`) — tracks active skill from `Skill` tool invocations and heuristics.
5. **Dashboard UI** (`servers/dashboard/public/index.html` + `servers/dashboard/src/ui/`) — single-page app connected via WebSocket.
6. **Log watcher fallback** (`servers/dashboard/src/adapters/log-watcher.ts`) — tails agent logs when hooks are unavailable.

## ADDED

### Runtime server

- New package/workspace `servers/dashboard`.
- Entry point: `crewloop dashboard` command (via `packages/cli` or `servers/dashboard/bin/`).
- HTTP `POST /event` endpoint accepts normalized events.
- WebSocket server on the same port broadcasts state snapshots and incremental events.
- In-memory session state aggregated per `session_id`.

### Agent instrumentation

- **Kimi Code adapter:** `[[hooks]]` entries for `~/.kimi-code/config.toml` calling `crewloop-shim kimi`.
- **Codex CLI adapter:** `~/.codex/hooks.json` entries calling `crewloop-shim codex`, plus documentation of the `[features] hooks = true` requirement.
- **OpenCode adapter:** plugin file `.opencode/plugin/crewloop-dashboard.js` that connects directly to the event server.
- **Log watcher:** optional filesystem tail for agent logs.

### Dashboard UI

- React/Vanilla JS single-page app.
- Components:
  - Active skill card (icon, name, status, agent source).
  - Skill activity graph/neural map.
  - Event timeline.
  - Session selector (when multiple sessions are active).
  - Telemetry panel (tool count, session duration, event rate).

### Skill metadata integration

- Reuse existing `packages/cli/src/resolver.ts` to parse `SKILL.md` frontmatter at server startup.
- Expose `/api/skills` endpoint returning skill name, description, and icon/emoji.

## MODIFIED

- `packages/cli/src/index.ts` (or equivalent) to add the `dashboard` subcommand.
- `README.md` to mention the dashboard and link to setup docs.
- `docs/` may later add a dashboard documentation page (owned by docs-writer).

## REMOVED

Nothing.

## Acceptance criteria

1. `crewloop dashboard` starts without errors and prints the local URL.
2. A WebSocket client connecting to the server receives the current sessions list and a full state snapshot.
3. Sending a `POST /event` with a normalized tool event updates the active session state and broadcasts to all clients.
4. The UI renders the active skill correctly when a `Skill` tool event is received.
5. The Kimi Code adapter config, when installed, causes tool events to appear in the dashboard.
6. The OpenCode plugin, when installed, causes tool events to appear in the dashboard.
7. The Codex adapter, when installed with `hooks = true`, causes Bash tool events to appear (file edit events are documented as best-effort).
8. No `tool_input` containing command strings, file contents, or MCP payloads is stored or broadcast.

## Event schema (normalized)

```typescript
interface DashboardEvent {
  id: string;                 // UUID generated by shim or server
  timestamp: number;          // epoch ms
  source: 'kimi' | 'codex' | 'opencode' | 'log-watcher';
  session_id: string;
  event_type:
    | 'session_start'
    | 'session_end'
    | 'tool_start'            // PreToolUse / tool.execute.before
    | 'tool_end'              // PostToolUse / tool.execute.after
    | 'skill_change';         // derived when Skill tool is invoked
  skill?: string;             // inferred active skill name
  tool?: string;              // tool name, e.g. 'Read', 'Bash', 'Skill'
  detail?: string;            // safe detail: file path, skill name, etc.
  status?: 'running' | 'success' | 'error';
  duration_ms?: number;
}
```

## Security requirements

- The shim MUST strip or redact `tool_input` fields except for allowlisted safe keys (`path`, `file_path`, `skill`, `subagent_type`).
- The server MUST bind to `127.0.0.1` by default.
- The server MUST NOT persist events to disk unless explicitly enabled.
- WebSocket messages MUST NOT include raw user prompts or tool outputs.
